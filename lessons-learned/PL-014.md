## Solutions

### The better-t-stack scaffold pre-ships most of the auth gate
`packages/api/src/index.ts` already had `protectedProcedure = publicProcedure.use(requireAuth)`
throwing `ORPCError("UNAUTHORIZED")`, `context.ts` already resolved+attached `session`,
and readers were already on `publicProcedure`. The PL-014 work was therefore surgical:
(1) extract a thin `getSession(headers)` helper into `packages/auth/src/index.ts` and have
`context.ts` consume it; (2) add the `web→api` contract message "Unauthorized console call"
to the existing 401; (3) write the verification tests. Resist the urge to rewrite working
scaffold code — rename churn would have broken `appRouter.privateData` for no benefit.

### oRPC UNAUTHORIZED maps to status 401 by default
`COMMON_ORPC_ERROR_DEFS.UNAUTHORIZED = { status: 401 }`. So `ORPCError("UNAUTHORIZED")`
already satisfies a "return 401" contract — assert both `err.code === "UNAUTHORIZED"` and
`err.status === 401` in tests; no custom status override needed.

### Test the gate with a handler-side sentinel
To prove "no console state is read/mutated when unauthenticated" (the spec's fail-closed
requirement), build the gated test procedure with a module-level `handlerRan` flag and assert
it stays `false` on the denied calls — proving the 401 short-circuits before the handler body,
not merely that an error was thrown.
