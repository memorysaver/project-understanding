## Solutions

### Adding a required context field churns sibling test helpers — make it optional
Adding `queue` to the oRPC `Context` as a *required* property broke every existing
router test's `ctx()` helper (posts/prompt/setPostStatus/auth), which build
`{ auth, session, db }`. Since production omits the field (enqueueDiscovery lazily
binds the real PIPELINE_QUEUE) and the orchestrator's `PipelineDeps.queue` is itself
optional, declaring `queue?: QueueProducer` on an explicit `Context` type (instead of
inferring it from a present-but-undefined literal) keeps the seam, matches prod
semantics, and touches zero sibling tests.

### routeTree.gen.ts: let the pre-commit oxfmt normalize it
The TanStack Router Vite plugin auto-regenerates `routeTree.gen.ts` in its own style
(single-quote/no-semicolon), which looks like a ~200-line diff vs the committed
double-quote/semicolon baseline. The repo's lefthook pre-commit runs
`oxfmt --write {staged_files}` with `stage_fixed: true`, so just `git add` the
regenerated file and commit normally — oxfmt reformats it to the canonical style and
the real diff collapses to only the new route lines.

### Casting context.db to the orchestrator's CrawlerDb without a new dep
`CrawlerDb` is exported from `@paperlens/crawler`, not re-exported by
`@paperlens/orchestrator`. To avoid adding a crawler dependency just for a type, cast
`context.db as PipelineDeps["db"]` (PipelineDeps IS exported) at the enqueueDiscovery
call boundary — same structural SQLite handle, nominally distinct.

## Summary for Next Agent

PL-020 was a clean thin-wrapper story: `triggerRun` is ~8 lines delegating to PL-018's
`enqueueDiscovery("manual", { db, queue })`. The only friction was the new optional
`queue?` context field (keep it optional to avoid churning sibling router tests) and
`routeTree.gen.ts` regeneration (let the pre-commit oxfmt normalize it — don't hand-edit).
The live workerd dev server was unreachable from the sandboxed shell (HTTP 000 on all
paths), so Tier-2 browser dogfood was waived and the behavior proven at the integration
layer instead. When you build the next console procedure (PL-016/PL-021/PL-022), the
context `queue` seam + the `run.test.ts` recording-fake-queue harness are reusable.
