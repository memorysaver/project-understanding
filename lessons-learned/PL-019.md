## Solutions

### Shared parseEntry helper preserves parseArxivAtom behavior
Factored the per-entry field extraction into a private `parseEntry(entry, id) =>
ArxivMetadata | null`. `parseArxivAtom` wraps it and throws on null (its tests
assert `.toThrow()` only, no message), while `parseArxivBatch` skips nulls. This
let the batch parser reuse the exact field/author/URL logic without touching the
single-id contract.

### batch id derived per-entry from <id>
The list feed returns recent submissions (not a known id), so each entry carries
its own `<id>http://arxiv.org/abs/2401.00011v2</id>`. `entryArxivId` strips the
`http(s)://arxiv.org/abs/` prefix and the trailing `vN` so the id matches the
arxiv_id dedup key.

### persist-only-new via pre-run snapshot
Snapshot the existing arxiv_ids (SELECT ... WHERE inArray(ids)) BEFORE the dedup
insert, then return the rows for `ids \ existing`. Both inArray calls are guarded
by non-empty length checks so no empty-IN query runs.

## Summary for Next Agent

Straightforward, additive library change. `discover` mirrors `fetchById`'s shape
(injected db + fetcher, ON CONFLICT DO NOTHING on arxiv_id) and reuses arxiv.ts.
MVP intentionally does NOT page — one list request returns the whole batch, so
etiquette is satisfied by a single USER_AGENT request; a future paged variant must
space requests by ARXIV_MIN_INTERVAL_MS (documented on `discover`). Orchestrator
rewiring (replacing DISCOVERY_SEED with a discover() call in handleDiscover) is a
SEPARATE follow-up — not done here. Tests are hermetic (fixture FetchLike +
in-memory bun:sqlite, no network, no sleep).
