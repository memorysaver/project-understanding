## Why

The public Reader API landed (PL-008) but there is no reader-facing web surface to
consume it. The reading capability needs its two Layer-0 pages: a chronological feed
of published posts and an article page for a single post. These are the minimal
functional pages — render the correct data with working navigation, no search/filter
and no visual polish (visual calibration is a later story, PL-012). This bundles
stories PL-009 (feed) and PL-010 (article), which ship together because they share
the generated TanStack Router route tree.

## What Changes

- Add the reader feed at `apps/web/src/routes/index.tsx` (replacing the scaffold
  placeholder): renders published posts newest-first via `api.listPosts`, each item
  linking to its article page. No search or filter UI.
- Add the article page at `apps/web/src/routes/posts.$id.tsx`: renders a post's
  title, sanitized body, and a link back to the source paper via `api.getPost`;
  handles the `NOT_FOUND` error (unknown or unpublished id) with a not-found state.
- The generated `apps/web/src/routeTree.gen.ts` regenerates on build to include the
  new `/posts/$id` route.

## Capabilities

### New Capabilities
- `reader-web`: the reader-facing web pages for the reading capability — a
  chronological feed of published posts and an article page (title, body, source
  link) — backed by the public Reader API, with a not-found state for hidden or
  missing posts.

## Impact

- `apps/web/src/routes/index.tsx` — chronological feed (PL-009).
- `apps/web/src/routes/posts.$id.tsx` — new article page (PL-010).
- `apps/web/src/routeTree.gen.ts` — regenerated by the build to register the route.
- No changes to packages, server, auth, DB schema, or product-context.yaml.
- No web test runner exists; correctness is proven via `bun run check-types` (which
  builds the web app) plus an independent review of the data wiring, with the route
  logic kept trivially correct.
